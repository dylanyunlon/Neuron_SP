## Overview

This document describes the code-generation evaluation framework introduced in
M821–M835.  The framework validates DES-LOC-trained models on three
complementary benchmarks and integrates periodic evaluation into the distributed
training loop.

---

## Benchmarks

### 1. HumanEval (pass@k)

| Property | Value |
|---|---|
| Source | `openai_humaneval` (HuggingFace Datasets) |
| Split | `test` (164 problems) |
| Metric | unbiased pass@k estimator (Chen et al. 2021) |
| k values | 1, 5, 10 |
| Samples/task | 10 completions |
| Temperature | 0.8 |

Each problem is attempted `n=10` times; the fraction that pass the unit test is
used to compute the unbiased pass@k via:

```
pass@k = 1 − C(n−c, k) / C(n, k)
```

where `c` = number of passing completions.

---

### 2. MBPP (basic programming)

| Property | Value |
|---|---|
| Source | `mbpp` (HuggingFace Datasets) |
| Split | `test` (~374 problems) |
| Metric | accuracy (fraction of tasks where the greedy completion passes all asserts) |
| Temperature | 0.2 (near-greedy) |

---

### 3. Commit Message Prediction *(DES-LOC signature task)*

| Property | Value |
|---|---|
| Source | CommitPackFT (`bigcode/commitpackft`, test split) |
| N samples | 1 000 |
| Input | unified diff (`<diff>…</diff>`) |
| Output | one-line commit message |
| Metrics | BLEU (bigram), ROUGE-L F1, exact match rate |
| Temperature | 0.1 |

This benchmark reflects the core DES-LOC training objective: learning to
summarise code changes as commit messages.

## Periodic Eval During Training

The eval harness is designed for heterogeneous GPU clusters:

```
Training cluster:          [ A100 × N ] [ H100 × 1 ]
                                ↓              ↓
At step % EVAL_INTERVAL:   continue       run_periodic_eval(...)
                           data-prep      (HumanEval + MBPP + Commit)
```

- Eval runs **only on the H100** (highest single-GPU throughput).
- All other GPUs continue tokenising / packing data for the next epoch.
- No training steps are skipped; overlap is near-complete for eval intervals ≥ 500 steps.

**Integration snippet** (add to training loop):

```python
from eval.run_eval import run_periodic_eval

EVAL_INTERVAL = 500   # adjust to balance overhead

if global_step % EVAL_INTERVAL == 0 and dist.get_rank() == H100_RANK:
    results = run_periodic_eval(
        model_path=checkpoint_dir,
        step=global_step,
        output_dir="desloc_results/eval_runs",
        benchmarks=["humaneval", "mbpp", "commit"],
    )
```

Results are written as `desloc_results/eval_runs/eval_step_XXXXXXX.json`.

## Quick Start

```bash
# dry-run with stubs (no model required)
python eval/run_eval.py --dry-run

# full eval against a checkpoint
python eval/run_eval.py \
    --model /path/to/desloc-checkpoint \
    --benchmarks humaneval mbpp commit \
    --output-dir desloc_results/eval_runs \
    --step 5000

# pre-download CommitPackFT test set for offline use
python eval/extract_commitpack_test.py --n 1000
```
